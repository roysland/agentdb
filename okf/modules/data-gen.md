## What it does

Provides generated data-access layer types and query methods for the agentdb system. It defines Go structs mapping to database tables (chunks, codebases, memories, workspaces, etc.) and query functions generated by sqlc that wrap raw SQL with type-safe Go signatures.

## Public interface

**Constructors:**
- `func New(db DBTX) *Queries` — constructs a `Queries` instance from any `DBTX`-compatible handle.
- `func (q *Queries) WithTx(tx *sql.Tx) *Queries` — returns a `Queries` bound to an active transaction.

**Query methods on `*Queries`:**
- `CreateChunk(ctx, CreateChunkParams) error`
- `CreateMemory(ctx, CreateMemoryParams) error`
- `DeleteChunksByCodebase(ctx, codebaseID int64) error`
- `DeleteMemoryByID(ctx, id string) (int64, error)`
- `GetChunksByCodebase(ctx, codebaseID int64) ([]Chunk, error)`
- `GetMemoryByID(ctx, id string) (Memory, error)`
- `ListCodebases(ctx) ([]Codebasis, error)`
- `ListMemoriesFiltered(ctx, ListMemoriesFilteredParams) ([]Memory, error)`
- `MarkMemoryRetrieved(ctx, MarkMemoryRetrievedParams) (int64, error)`
- `RegisterCodebase(ctx, RegisterCodebaseParams) (int64, error)`
- `SearchMemoriesLexicalFiltered(ctx, SearchMemoriesLexicalFilteredParams) ([]Memory, error)`
- `UpdateMemory(ctx, UpdateMemoryParams) (int64, error)`

**Key types:** `DBTX` (interface), `Queries`, `Chunk`, `Codebasis`, `Memory`, `Workspace`, `WorkspaceMember`, `IndexedFile`, `SourceFile`, `Symbol`, `Edge`, `ChunksFt`, `CodebaseMetum`, `Meta`, and all `*Params` structs.

## Key invariants

- `CreateChunk` uses `ON CONFLICT(codebase_id, chunk_key) DO UPDATE`, making `(codebase_id, chunk_key)` an upsert key — callers must ensure `chunk_key` is deterministic for a given logical chunk.
- `RegisterCodebase` returns `LastInsertId()`, so the database must use integer autoincrement primary keys for this to be meaningful.
- `DeleteMemoryByID` and `MarkMemoryRetrieved` return `RowsAffected()`, meaning callers can distinguish "not found" from "no-op" by checking the count.
- `ListMemoriesFiltered` and `SearchMemoriesLexicalFiltered` use sentinel empty-string / NULL interface parameters to conditionally apply filters — a `NULL` in `WorkspaceID` or `CodebaseID` means "no filter," while `''` in `Category` means "no filter."
- `Memory` tracks `RetrievalCount` and `LastRetrieved` as nullable fields, implying a memory may never have been retrieved.
- `Edge.Resolved` is `int64` (not bool), suggesting it stores a status code or timestamp rather than a simple resolved/unresolved flag.

## Non-obvious decisions

- **`ChunksFt` struct** — a separate type with only `Snippet`, `Name`, and `FilePath` fields, not mapped to any table in `models.go`. This strongly suggests it's a virtual or full-text index table (e.g., SQLite FTS5 shadow table), but without the schema file its exact role is inferred rather than certain.
- **`SearchMemoriesLexicalFiltered` uses `LIKE` for search** — this is a lexical (substring) search, not FTS, despite the presence of `ChunksFt` which implies full-text indexing exists for chunks. Memories are intentionally searched with simple pattern matching.
- **`Edge.TargetCodebaseID` is nullable** — edges can reference symbols in a different codebase, and `NULL` means the edge is intra-codebase or unresolved across codebases.

## Unclear intent

- **`Meta` struct** — defined in `models.go` with `Key`/`Value` fields but no corresponding query or table reference in `queries.sql.go`. Its purpose and which table it maps to cannot be determined from this module alone.
- **`CodebaseMetum`** — the unusual Latinized plural (`Metum` instead of `Meta`) suggests intentional naming, but without schema or usage context it's unclear whether this is a distinct concept from `Meta` or a naming accident.
- **`ChunksFt`** — as noted above, the exact table type (FTS shadow table, materialized view, etc.) cannot be confirmed without the SQL schema.
